Why Static Sites Are Awesome

A short essay on choosing simplicity over complexity

I recently put together a tool to find cheap mini PCs. It started as a Python script that pulled data from eBay, did a bunch of parsing, and printed the results to the terminal. I figured, “Hey, maybe someone else would find this useful.” So why not turn it into a website?

My first thought: Let’s use Flask! Python in the back, maybe some Svelte for a reactive front-end… and before I knew it, I was getting overwhelmed. Then I remembered a site I’d seen a while ago, diskprices.com, that compared Amazon listings for storage. I peeked at the source code, and to my surprise, it was just a single HTML page with no backend. I thought, “Wait… can I do that too?”

I closed all my tabs about the “best web frameworks,” shut down my computer, and stepped away. I came back the next day, determined to keep things simple. Just get it done.

After a few rounds of “is there anything simpler?” I landed on the obvious answer: a static HTML page that displays pre-fetched data. Run the Python script periodically, save the data to a JSON file, and load it into a static page. Vanilla JavaScript handles sorting and filtering.

Hosting? Cloudflare Pages — free. Updating the data? GitHub Actions on a cron schedule — also free. The next morning everything was working. The data updated every hour and it just… worked.

I submitted it to Hacker News and it hit the front page. 275k requests that day. Cloudflare handled it without breaking a sweat. All for free.

That’s the beauty of static sites. They’re simple, fast, and get the job done without the headaches. Sometimes, simpler is better.
